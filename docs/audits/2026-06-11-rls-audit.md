# RLS Security Audit — trainers.gg

**Date:** June 11, 2026
**Method:** live `pg_policies` dump from local Supabase (full migration replay), analyzed through three independent lenses (over-read surface, coverage/hygiene, architecture fitness).
**Scope:** 64 public tables, 175 effective policies, 80 SECURITY DEFINER functions, 6 realtime-published tables.

> Migration-file analysis was rejected as a method (488 `CREATE POLICY` vs 471 `DROP POLICY` churn across 235 migrations) — the live database is the source of truth.

## Executive summary

| Lens                       | Grade  | Summary                                                                                 |
| -------------------------- | ------ | --------------------------------------------------------------------------------------- |
| Coverage & hygiene         | **B+** | RLS on 100% of tables; initplan pattern everywhere; explicit deny-alls on system tables |
| Over-read / scrape surface | **C+** | Risk is real but concentrated in ~5 shared tables, not systemic                         |
| Architecture fitness       | —      | Per-user tables provably self-scoping; shared tables are the entire scrape surface      |

**Strengths:** zero bare `auth.uid()` (every policy uses the initplan `(SELECT auth.uid())` form); RLS enabled on all 64 tables; `rate_limits` / `atproto_oauth_state` explicitly deny-all; zero direct browser table-writes in the app (all writes via Server Actions / SECURITY DEFINER RPCs).

## Critical findings

### 1. `users` table is world-readable — CRITICAL

- **Policy:** `SELECT USING (true)` — anon + authenticated
- **Exposes:** `email`, `phone_number`, `birth_date`, `first_name`, `last_name`, `external_accounts` (OAuth JSON), `last_sign_in_at`, `is_locked`
- **Impact:** trivial bulk email/PII harvest by unauthenticated users; GDPR/compliance exposure. `alts.user_id` (also public) maps alt → user → email.

### 2. `tournament_team_sheets` has no phase gate — CRITICAL (competitive integrity)

- **Policy:** `SELECT USING (true)` — anon + authenticated
- **Exposes:** full OTS — `species`, `ability`, `held_item`, `move1-4`, `nature`, `tera_type` — for any tournament at any time
- **Impact:** scrape opposing team sheets mid-tournament, before reveal. The "only after tournament starts" check exists only in app code (`getTournamentTeamSheets()`); a direct PostgREST call bypasses it. The match page even documents the hole: "UI-level visibility toggle (NOT an access-control guarantee)" (`r/[round]/t/[table]/page.tsx`).

### 3. `tournament_registrations` leaks staff-internal fields — HIGH

- **Policy:** `SELECT USING (true)`
- **Exposes:** `drop_notes` (free-text staff notes on dropped players), `rental_team_photo_key`/`url` (storage keys — and the `uploads` bucket is **public**, so the images themselves are fetchable), `dropped_by` (staff UUID)

### 4. Dead realtime subscription — BUG

- App subscribes to `tournament_invitations` changes, but the table is **not in the `supabase_realtime` publication** → events never deliver, silently.

### 5. Anon EXECUTE on privileged RPCs — MEDIUM

- `admin_alter_cron_schedule`, `admin_get_cron_schedules`, `register_for_tournament_atomic`, `accept_tournament_invitation_atomic`, `send_tournament_invitations_atomic`, `custom_access_token_hook` are executable by `anon`.
- All have internal guards (no live exploit found), but it is needless attack surface. Root cause: Postgres grants `EXECUTE` to `PUBLIC` on every new function by default.

## Realtime alignment

| Table (in publication)                                                | Bucket | RLS for subscribers               | Status                                  |
| --------------------------------------------------------------------- | ------ | --------------------------------- | --------------------------------------- |
| `notifications`                                                       | P      | `user_id = auth.uid()`            | OK — ideal                              |
| `match_games`, `match_messages`                                       | R      | participant-or-staff (4-hop join) | OK, costly per event                    |
| `tournament_matches`, `tournament_registrations`, `tournament_rounds` | S      | `true`                            | OK                                      |
| `tournament_invitations`                                              | P      | —                                 | **Subscribed but not published — dead** |

These tables must stay client-readable (or move to Broadcast) under any architecture, since `postgres_changes` requires the subscriber to pass RLS.

## Architecture conclusion

- Per-user (P) direct reads are safe — the quals are structurally self-scoping.
- The over-scraping concern is real but lives entirely in S — which a cached API layer addresses.
- R needs case-by-case treatment — flatten the hot quals or API-route the admin clusters.
