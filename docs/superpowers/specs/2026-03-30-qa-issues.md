## Critical

### 1. Alt Switcher Does Not Function as Global Filter

**Location:** `apps/web/src/components/dashboard/dashboard-sidebar.tsx`

**Problem:** The alt switcher currently acts as a **navigation shortcut** — selecting an alt navigates to `/dashboard/alts/[username]`, and clicking any other sidebar link (Home, Tournaments, etc.) resets to the main alt. Per product decision, the alt switcher should be a **global filter** that persists across all dashboard views.

**Current behavior:**

1. User selects `admin_trainer_vgc` in alt switcher
2. Navigates to `/dashboard/alts/admin_trainer_vgc`
3. Clicks "Home" → `/dashboard` resets to main alt (`admin_trainer`)
4. All context of the selected alt is lost

**Expected behavior:**

1. User selects `admin_trainer_vgc` in alt switcher
2. Selection persists across all dashboard pages
3. `/dashboard` shows data filtered to `admin_trainer_vgc`
4. `/dashboard/tournaments` shows tournaments for `admin_trainer_vgc`
5. Selecting "All Alts" returns to aggregate view

**Implementation requirements:**

- Persist selected alt via cookie (preferred for SSR compatibility) or localStorage
- Dashboard pages read the stored selection and filter data accordingly
- "All Alts" = aggregate mode (current `/dashboard` default behavior)
- Sidebar navigation does not reset selection
- URL stays clean (`/dashboard`, not `/dashboard?alt=admin_trainer_vgc`)

**Root cause:** The spec says "State derived from URL path, not client state" — this was interpreted literally, so the selection only exists when the username is in the URL. The spec needs updating to reflect the global filter decision.

### 2. Alt Switcher Subtitle Repeats Username for Non-Main Alts

**Location:** `apps/web/src/components/dashboard/dashboard-sidebar.tsx:220`

**Problem:** When viewing a non-main alt, the trigger subtitle repeats the alt username. Name and subtitle are identical.

**Current behavior:**
| Context | Name | Subtitle | Status |
|---------|------|----------|--------|
| `/dashboard/alts` | admin_trainer | "All alts" | Correct |
| `/dashboard/alts/admin_trainer` | admin_trainer | "Main alt" | Correct |
| `/dashboard/alts/admin_trainer_vgc` | admin_trainer_vgc | "admin_trainer_vgc" | Bug |

**Root cause (line 216-220):**

```typescript
const triggerSubtitle = isAllAltsPage
  ? "All alts"
  : selectedAlt?.isMain
    ? "Main alt"
    : (selectedAlt?.username ?? "All alts"); // Falls back to username
```

**Note:** This issue will likely be resolved as part of the global filter rework (issue #1). Once the alt switcher persists state, the subtitle logic needs to handle: "Main alt", "All alts", and a label for non-main alts (e.g., "Secondary alt" or "Alt").

### 5. Tournament Manage Page Layout Broken in New Dashboard Shell

**Location:** `apps/web/src/app/(dashboard)/dashboard/community/[communitySlug]/tournaments/[tournamentSlug]/manage/page.tsx`

**Problem:** The tournament manage page has a dark/black background in the main content area. The page is a thin Server Component wrapper that imports `TournamentManageClient` from the **old** `(app)/to-dashboard/` route group. That old component was designed for a different layout shell and doesn't match the new dashboard conventions.

**Root cause:**

1. The manage page at the new dashboard route delegates entirely to:
   `apps/web/src/app/(app)/to-dashboard/[communitySlug]/tournaments/[tournamentSlug]/manage/tournament-manage-client.tsx` (359 lines)
2. The old component has **no `PageHeader`** — it uses its own back-arrow + title pattern
3. It renders a `<div>` with a gradient fade referencing `from-background`, which likely resolves to a dark color in the new dashboard shell context
4. The component assumes the old layout's background/spacing, causing visual breakage

**Screenshot:** User-reported — dark/black background covering the main content area at `/dashboard/community/vgc-league/tournaments/vgc-league-championship-week-02/manage`

**Fix approach:** The `TournamentManageClient` component needs to be either:

- Migrated into the new dashboard route group with proper `PageHeader` and layout conventions, or
- Wrapped with appropriate overrides to work within the new shell

## Spec Clarification Needed

### Alt Switcher as Global Filter (decided)

The spec originally said:

> State derived from URL path, not client state

Product decision (2026-03-30): The alt switcher is a **global filter**, not a navigation shortcut. This means:

- Selection must persist across page navigation (cookie/localStorage)
- All dashboard views filter by the selected alt
- "All Alts" is the aggregate/default mode
- URL does not need to contain the alt username

**The spec (`2026-03-30-dashboard-redesign-v2.md`) should be updated to reflect this decision.**
